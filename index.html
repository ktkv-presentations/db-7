Концепция ER-диаграмм

ER-модель

концептуальная модель данных, которая описывает структуру базы данных с помощью графического представления сущностей, их атрибутов и взаимосвязей

						erDiagram
    CUSTOMER ||--o{ ORDER : ""
    CUSTOMER {
        string name
        string custNumber
        string sector
    }
    ORDER ||--|{ LINE-ITEM : contains
    ORDER {
        int orderNumber
        string deliveryAddress
    }
    LINE-ITEM {
        string productCode
        int quantity
        float pricePerUnit
    }
    ORDER ||--|{ STUDENT : contains
    STUDENT {
        int orderNumber
        string deliveryAddress
    }

Цель ER-модели

упрощения проектирования баз данных, позволяя разработчикам визуализировать данные и их отношения

История создания ER-модели

Разработана Питером Ченом в 1976 году как способ формализации представления данных для улучшения процесса проектирования баз данных

Широко используется в различных системах управления базами данных (СУБД)

Компоненты ER-модели

Сущности

объекты, представляющие реальные вещи или понятия, которые имеют уникальное существование в системе

Преимущества ER-модели

Упрощение визуализации данных

Графическое представление облегчает понимание структуры базы данных и ее логики

Четкость и однозначность

ER-диаграммы помогают избежать неоднозначности в описании требований к данным

Поддержка совместной работы

Упрощает коммуникацию между разработчиками и заинтересованными сторонами, позволяя легко обсуждать изменения и улучшения

Виды атрибутов

Простые атрибуты

Не могут быть разделены на более мелкие компоненты

Составные атрибуты

Могут быть разделены на более мелкие компоненты, которые имеют самостоятельное значение

Многоценные атрибуты

Могут содержать несколько значений или экземпляров

Выводные атрибуты

Значение атрибута может быть вычислено на основе других атрибутов

Атрибуты как ключи

Первичный ключ

Уникальный идентификатор для каждой сущности, который позволяет однозначно отличать одну запись от другой

Внешний ключ

Атрибут, который связывает одну сущность с другой, указывая на первичный ключ другой сущности

Важность атрибутов в проектировании

  • Атрибуты как основа данных - Правильное определение атрибутов помогает в проектировании базы данных, которая отвечает потребностям бизнеса
  • Качество данных - Четкое описание атрибутов позволяет избежать неясностей и ошибок в хранении и обработке данных
  • Оптимизация запросов - Знание атрибутов помогает разработать эффективные SQL-запросы для выборки, обновления и удаления данных

Кардинальности (Cardinality)

Кардинальность определяет количество сущностей, которые могут участвовать в конкретной связи

Один к одному (1:1)

одна сущность связана только с одной другой сущностью

Один ко многим (1)

одна сущность может быть связана с множеством других

Многие ко многим (M)

множество сущностей с одной стороны может быть связано с множеством сущностей с другой стороны

Визуальная нотация кардинальности

ERD в нотации Чена

Сущности

  • Представляются в виде прямоугольников
  • Название сущности записывается внутри прямоугольника

ERD в нотации Чена

Связи

  • Представляются в виде ромбов
  • Название связи записывается внутри ромба, и она соединяет сущности линиями

ERD в нотации Чена

Атрибуты

  • Представляются в виде ромбов
  • Название связи записывается внутри ромба, и она соединяет сущности линиями
